# AI 面试官 Agent 设计思路

## 一、项目愿景

构建一个可替代人类面试官的 AI Agent，能够自主完成技术面试的全流程：从开场破冰、技术问答、编程考核，到总结评分。目标是提供标准化、公平、高效的面试体验。

---

## 二、核心功能模块

### 2.1 面试流程引擎（状态机驱动）

整个面试采用有限状态机（FSM）管理，确保流程可控、可回溯：

```
[等待候选人] → [开场/破冰] → [简历深挖] → [技术问答轮次]
    → [编程考核(可选)] → [候选人反问] → [结束/评分]
```

每个阶段有明确的：
- **进入条件**：前置阶段完成
- **最大时长**：超时自动推进或友好提示
- **退出条件**：问题问完 / 候选人跳过 / 超时
- **回退能力**：支持回到上一题（候选人要求澄清时）

### 2.2 知识库与题库系统

```
题库结构：
├── 技术栈分类（前端/后端/算法/AI/DevOps/...）
│   ├── 初级 / 中级 / 高级
│   ├── 选择题、简答题、场景题
│   └── 编程题（LeetCode 风格 + 实战项目）
├── 行为面试题库（STAR 方法）
│   ├── 团队协作
│   ├── 冲突处理
│   ├── 项目管理
│   └── 职业规划
└── 岗位定制模板
    ├── 后端工程师.json
    ├── 前端工程师.json
    └── 数据分析师.json
```

每道题目包含：
- `id`, `category`, `difficulty`, `question_text`
- `expected_keywords`：期望答案中的关键词
- `scoring_rubric`：评分标准（1-5 分 / 多维评分）
- `follow_up_questions`：追问列表
- `time_limit_seconds`：建议答题时长

### 2.3 对话管理（Conversation Manager）

```
┌──────────────────────────────────────┐
│          ConversationManager          │
│                                      │
│  ┌──────────┐  ┌──────────────────┐ │
│  │ 上下文窗口 │  │  对话历史记录     │ │
│  │ (滑动窗口) │  │  (完整持久化)    │ │
│  └──────────┘  └──────────────────┘ │
│                                      │
│  ┌──────────┐  ┌──────────────────┐ │
│  │ 意图识别  │  │  情绪/状态感知    │ │
│  │ (跳过/澄 │  │  (紧张/自信/回避) │ │
│  │  清/深挖)│  └──────────────────┘ │
│  └──────────┘                       │
└──────────────────────────────────────┘
```

关键设计：
- **滑动窗口上下文**：只保留最近 N 轮对话 + 系统指令，避免 token 溢出
- **结构化历史**：每轮对话存为 `{speaker, content, timestamp, phase, metadata}`
- **意图识别**：判断候选人是想跳过、需要澄清、还是在回避问题

### 2.4 评估引擎

```
评估维度：
┌─────────────────────────────────────────────────────┐
│ 维度           │ 权重  │ 评估方式                    │
├─────────────────────────────────────────────────────┤
│ 技术准确性      │ 30%  │ 关键词匹配 + LLM 语义评判    │
│ 思维深度/广度   │ 20%  │ LLM 分析回答层次            │
│ 沟通表达能力    │ 15%  │ 清晰度、结构化程度           │
│ 问题解决能力    │ 20%  │ 编程题通过率 + 思路分析      │
│ 行为匹配度      │ 15%  │ STAR 完整性评分             │
└─────────────────────────────────────────────────────┘
```

评估流程：
1. **单题评分**：每答完一题即时评估（延迟 < 2 秒）
2. **追问决策**：根据评分决定是追问、进入下一题还是跳过
3. **综合评分**：面试结束后汇总加权总分
4. **报告生成**：输出结构化面试报告（优点/不足/建议/是否推荐）

### 2.5 编程考核模块

```
┌────────────────────────────────────────┐
│          Code Assessment Module         │
│                                        │
│  ┌──────────┐    ┌──────────────────┐ │
│  │ 题目展示  │ →  │  代码编辑器/IDE   │ │
│  └──────────┘    └────────┬─────────┘ │
│                           ↓            │
│  ┌──────────┐    ┌──────────────────┐ │
│  │ 测试判题  │ ←  │  代码提交         │ │
│  └──────────┘    └──────────────────┘ │
│       ↓                               │
│  ┌──────────────────────────────────┐ │
│  │ AI 代码审查 (复杂度/风格/边界)    │ │
│  └──────────────────────────────────┘ │
└────────────────────────────────────────┘
```

- 集成代码执行沙箱（Docker 隔离）
- 支持多语言：Python / JavaScript / Java / Go / C++
- 不仅判断通过/失败，还分析：
  - 时间复杂度 & 空间复杂度
  - 代码风格与可读性
  - 边界条件处理
  - 解题思路演进（是否卡住后自行调整）

### 2.6 多模态交互

| 模式 | 适用场景 | 实现方式 |
|------|----------|----------|
| **纯文本** | 文字聊天面试 | WebSocket + Markdown 渲染 |
| **语音输入** | 口语回答 | ASR（Whisper / Azure Speech）→ 文本 |
| **语音输出** | 提问播报 | TTS（Edge TTS / ElevenLabs） |
| **视频** | 表情/神态分析 | 可选，用于反作弊 + 情绪分析 |

### 2.7 反作弊与公平性保障

- **切屏检测**：前端监听页面可见性变化
- **复制粘贴监控**：编程环节限制粘贴或标记大段粘贴
- **回答一致性检查**：检测候选人是否在不同轮次给出矛盾回答
- **题库随机化**：同一岗位抽取不同题目组合，保证公平
- **标准化评分**：同一维度的评分标准固定，减少主观偏差

### 3.2 技术栈推荐

| 层 | 技术选择 | 理由 |

| **LLM 调用** | Anthropic SDK / LiteLLM | 多模型可插拔、统一接口 |
| **数据库** | PostgreSQL + SQLAlchemy | 结构化面试数据 |
| **缓存** | Redis | 对话状态缓存、限流 |

| **代码沙箱** | Docker + gVisor | 安全隔离代码执行 |
| **语音** | Whisper (本地) / Azure Speech | 成本可控 |
| **前端** | Next.js + Monaco Editor | 代码编辑器 + 现代化 UI |
| **部署** | Docker Compose / K8s | 可扩展 |

## 四、核心 Prompt 设计思路

### 4.1 系统级 System Prompt

```
你是一个经验丰富的技术面试官。你的特质：
- 专业但不冷漠，温暖但不随意
- 根据候选人的水平动态调整题目难度
- 善于追问，挖掘候选人真实水平
- 严格但有同理心，候选人卡住时给予适当引导
- 严格遵守面试流程，不跳过关键环节

当前面试上下文：
- 岗位：{job_title}
- 年限要求：{experience_level}
- 重点考察：{key_skills}
- 面试阶段：{current_phase}
- 已用时间：{elapsed_time}
```

### 4.2 评估 Prompt

```
请对以下候选人的回答进行多维度评估：

【题目】{question}
【期望关键词】{expected_keywords}
【候选人回答】{candidate_answer}

请输出 JSON：
{
  "score": 1-5,
  "keyword_coverage": ["命中的关键词"],
  "depth": "shallow | moderate | deep",
  "clarity": 1-5,
  "missed_points": ["遗漏的知识点"],
  "follow_up_suggestion": "追问建议（或 null）",
  "overall_comment": "一句话评价"
}
```

### 4.3 动态难度调整策略

- 连续答对 2 题 → 自动升级到高一级难度的题目
- 连续答错 2 题 / 明显卡住 → 降级或给予提示
- 回答质量一般 → 同级追问，不急于升级
- 达到目标难度后稳定出题，验证是否真正掌握

## 五、关键设计决策点

### 5.1 "替代面试官"到什么程度？

| 级别 | 描述 | 人类参与 |
|------|------|----------|
| **辅助模式** | AI 出题 + 初评，人类面试官终审 | 高 |
| **半自动** | AI 全程面试，人类只看报告决定是否通过 | 中 |
| **全自动** | AI 面试 + 评分 + 决策，人类只处理申诉 | 低 |

建议先从**辅助模式**起步，积累数据和信心后再逐步自动化。

### 5.2 如何处理边界情况？

| 场景 | 处理策略 |
|------|----------|
| 候选人回答完全偏题 | 礼貌纠正，给一次重新回答的机会 |
| 候选人情绪崩溃/放弃 | 暂停面试，询问是否需要休息 |
| 候选人反问答案 | 不直接给答案，引导思考方向 |
| 候选人请求跳过 | 记录跳过次数，超过阈值标记为异常 |
| 网络中断 | 支持断线重连，从中断点恢复 |
| LLM 输出异常 | Fallback 模板问题 + 人工接管告警 |

### 5.3 面试公平性考量

- **题目难度标准化**：每个知识点有明确的难度标签和校准机制
- **避免 LLM 偏见**：定期审计评分分布，检查是否对某些群体存在系统性偏差
- **候选人知情权**：面试开始前告知流程、时长、评分维度（但不说具体题目）
- **申诉渠道**：候选人可对评分提出异议，进入人工复核

## 六、MVP 建议

### 第一阶段（2-4 周）

- [ ] 纯文本面试（WebSocket 实时对话）
- [ ] 基础状态机（开场 → 技术问答 → 结束）
- [ ] 静态题库（20 道题，前端/后端各半）
- [ ] 单维评分（1-5 分 + 简短评语）
- [ ] 面试记录持久化 + 简单回放

### 第二阶段（4-6 周）

- [ ] 编程考核模块（代码编辑器 + 沙箱执行）
- [ ] 动态难度调整
- [ ] 追问能力增强
- [ ] 综合评分 + 报告生成
- [ ] 题库管理后台

### 第三阶段（6-10 周）

- [ ] 语音输入/输出
- [ ] 简历解析 + 定制化面试
- [ ] 反作弊机制
- [ ] 多对一面试（多个 AI 面试官角色）
- [ ] 数据分析仪表盘

## 七、风险与挑战

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| LLM 幻觉导致错误评判 | 高 | 关键评分点使用关键词匹配兜底；人工抽检 |
| 候选人抵触 AI 面试 | 中 | 透明告知流程；保留人类复核通道 |
| 题库泄露 / 作弊 | 中 | 题目动态组合；反作弊监控 |
| API 延迟影响体验 | 低 | 流式输出 + 打字机效果；本地模型作为 fallback |
| 法规合规（数据隐私） | 高 | 数据加密存储；候选人可要求删除数据；遵循 GDPR/个保法 |

## 八、竞品 / 参考

- **CodeSignal / HackerRank**：偏重编程测试，缺乏对话交互
- **Karat**：真人远程面试，我们想替代的是这个环节
- **Byteboard**：Google 孵化的异步技术面试平台
- **TalkBerry**：AI 口语面试（侧重语言能力）
- 本项目定位：**异步/实时可选的、多模态的、覆盖软硬技能的综合技术面试 Agent**

---

> 💡 **核心哲学**：AI 面试官的目标不是"比人类更聪明"，而是"比人类更一致、更耐心、更无偏见"。把结构化、重复性的评估工作交给 AI，让人类面试官专注于那些真正需要直觉和同理心的判断。
